pi를 위한 Agentic Coding Harness

요청을 명확히 하고, 실행 가능한 계획으로 바꿔서, 안전하게 구현하세요.

oh-my-jinho는 pi에 실전형 워크플로우를 추가해 모호한 요청을 Context Brief와 실행 계획으로 정리해줍니다.

바로 시작 저장소 보기

1) 설치

GitHub에서 바로 설치:

pi install git:alphabeen/oh-my-jinho

버전 고정 설치:

pi install git:alphabeen/oh-my-jinho#v1.0.0

2) 빠른 시작 (3단계)

  1. 설치 후 /setup을 1회 실행하세요 (quiet startup 설정).
  2. /clarify [주제]로 Context Brief를 먼저 만드세요.
  3. /plan [주제]로 실행 가능한 구현 계획을 생성하세요.
Tip: 작업이 끊겨도 /resume으로 워크스페이스별 세션을 이어갈 수 있습니다.

3) 5분 체크리스트

  • pi install 완료
  • /setup 실행 완료
  • /clarify로 범위 확정
  • /plan으로 실행 계획 확보

4) 이런 점이 좋아요

  • 읽기 쉬운 구조: 짧은 TypeScript와 Markdown 중심이라 동작 흐름을 코드로 직접 확인하기 쉽습니다.
  • 실시간 상태 확인: footer에서 캐시 히트율, 컨텍스트 사용량, 현재 실행 상태를 바로 확인할 수 있습니다.
  • 서브에이전트 실행: subagent를 별도 pi 프로세스로 띄워 단일 / 병렬 / 체인 모드로 실행할 수 있습니다.
  • 내장 스킬 12개: clarification, plan crafting, milestone planning, run plan, review, debugging 등 실무 흐름을 지원합니다.

실무형 스킬 안내 (Option 3)

추천 흐름: clarify → plan → systematic-debugging → simplify → review

1) agentic-clarification

명확화
  • 언제 요구사항이 모호할 때
  • 산출물 실행 가능한 Context Brief
  • 핵심 질문 1개씩 + 병렬 탐색으로 범위 고정

2) agentic-plan-crafting

계획
  • 언제 구현 전에 작업 순서를 확정할 때
  • 산출물 의존성/검증이 명시된 실행 계획
  • 핵심 플레이스홀더 없는 태스크 단위 계획

3) agentic-systematic-debugging

디버깅
  • 언제 버그/테스트 실패 원인 분석이 필요할 때
  • 산출물 재현-원인-수정-검증 체인
  • 핵심 원인 고정 전 섣부른 수정 금지

4) agentic-simplify

정리
  • 언제 변경 후 코드 품질/중복을 정리할 때
  • 산출물 리뷰 반영된 간결한 변경셋
  • 핵심 병렬 리뷰 후 최소 수정으로 정리

5) agentic-review-work

검증
  • 언제 구현 완료 후 독립 검증이 필요할 때
  • 산출물 PASS/FAIL 리뷰 문서
  • 핵심 실행 맥락 없이 계획/코드만으로 검증

추천 사용 순서

  1. /clarify — 모호한 요청을 Context Brief로 고정
  2. /plan — 파일/의존성/검증 기준을 먼저 잠금
  3. systematic-debugging — 이슈가 생기면 재현→원인→수정 순서 준수
  4. simplify — 변경분 품질/중복 정리
  5. review-work — 독립 PASS/FAIL 검증으로 마무리
실패 방지 팁: 요청이 애매하면 바로 구현하지 말고 /clarify부터 시작하세요. 그다음 /plan으로 파일/검증 범위를 잠그면 재작업을 크게 줄일 수 있습니다.
상세: Clarification 구현 포인트
  • 한 번에 질문 하나
  • 코드베이스 병렬 탐색 결과를 Context Brief로 통합
  • 모호성 해소 전 구현 금지
상세: Plan Crafting 구현 포인트
  • 정확한 파일 경로와 명령 포함
  • 병렬 가능/의존 관계 명시
  • 최종 검증 태스크 필수
상세: Debugging / Simplify / Review 포인트
  • Debugging: 재현 → 증거 → 단일 원인 가설 → 검증
  • Simplify: 변경분 중심 재사용/품질/효율 정리
  • Review: 독립 시점 PASS/FAIL로 완료 품질 확인

핵심 커맨드

커맨드설명
/clarify [주제]요청을 질문 기반으로 명확화하고 Context Brief를 만듭니다.
/plan [주제]실행 가능한 구현 계획을 만듭니다.
/ultraplan [주제]복잡한 작업을 마일스톤 DAG로 분해합니다.
/resume이전 세션으로 돌아가 작업 흐름을 이어갑니다.
/mode현재 IDS 모드 상태와 컨텍스트 파일/워크플로우 스냅샷을 확인합니다.
/reset-phase워크플로우 상태를 idle로 초기화합니다.

컨텍스트 주입 방식

프로젝트 루트(또는 상위 디렉토리)에서 아래 순서로 컨텍스트 파일을 자동 탐색합니다.

  1. IDS.md (기본 우선)
  2. OMJ.md (하위 호환)
  3. PROJECT_CONTEXT.md
  4. AGENT_CONTEXT.md

탐색된 파일은 매 턴 다시 읽어 최신 팀 규칙을 반영합니다.

IDS Soft Mode: IDS.md가 있으면 IDS 전용 가이던스가 자동 활성화됩니다. 파일이 없으면 일반 모드로 동작하지만 IDS 규칙 적용을 위해 IDS.md를 권장합니다.

장기 메모리

oh-my-jinho는 공용 장기 메모리를 사용해 이전 세션의 Context Brief, 결정사항, 최근 요약을 다시 불러옵니다. 저장 데이터는 사용자 홈 디렉토리의 ~/.pi/omj-memory.db에 보관됩니다.

  • memory_write — Context Brief나 결정사항을 마크다운으로 저장
  • memory_search — 과거 세션 요약과 세션 메타데이터 검색
  • memory_recent — 최근 세션과 최근 요약 확인
  • memory_recall — 특정 세션 ID로 상태와 최근 맥락 회수
추천 흐름: /clarify로 Context Brief를 만든 뒤 memory_write로 저장하고, 필요할 때 memory_search, memory_recent, memory_recall로 다시 불러오세요.

FAQ

매번 --extension 경로를 직접 지정해야 하나요?

아니요. pi install 이후에는 pi가 설치 경로를 관리합니다.

코드를 자동으로 막 수정하나요?

명시적으로 write/edit가 실행될 때만 파일이 바뀝니다. 기본은 명확화와 계획 수립부터 진행합니다.

subagent는 어떤 방식으로 실행되나요?

subagent는 별도 pi 프로세스로 실행됩니다. single(단일 조사), parallel(독립 작업 동시 실행), chain(이전 결과를 다음 단계로 전달) 모드를 선택해 오케스트레이션합니다.

/resume은 어디까지 복원하나요?

세션 전환과 함께 워크플로우 상태(phase, 최근 문서/주제, 중단 여부)를 복원합니다. 필요한 경우 이어가기 프롬프트를 자동으로 제안합니다.

footer 지표는 어떻게 읽나요?

cache는 프롬프트 캐시 적중률, ctx는 컨텍스트 사용량, active tools는 현재 실행 중인 도구 상태를 의미합니다.

내장 스킬은 무엇을 도와주나요?

현재 내장 스킬은 12개이며, 요구사항 명확화부터 계획 수립, 실행, 디버깅, 리뷰/정리까지 실무 흐름을 단계별로 지원합니다.

재시작했는데 원점으로 돌아간 것 같아요. 무엇을 확인하나요?

먼저 /mode를 실행해 IDS 모드 상태와 Context file 경로를 확인하세요. 그다음 현재 실행 위치(cwd)가 프로젝트 루트인지 점검하세요.

다른 확장/동일 스킬 이름과 충돌할 수 있나요?

가능합니다. 동일 이름 스킬이 여러 확장에서 로드되면 우선순위가 엇갈릴 수 있으니, 충돌이 의심되면 중복 스킬을 비활성화하고 단일 하네스로 운영하세요.
